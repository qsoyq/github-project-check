# GitHub 研发工作流介绍

> 版本：v0.1  
> 状态：草案  
> 定位：本文件描述从需求立项到线上发布、复盘沉淀的完整研发工作流。  
> 相关文档：[总纲](./github.md) / [研发协作规范](./github-standard.md) / [自动化工具介绍](./github-automation.md)

---

## 1. 工作流目标

本工作流的目标是把研发过程中的需求、任务、代码、验证、审查、发布和复盘串成一个可追溯的闭环。

最终希望达成：

1. 每个需求都有明确来源。
2. 每个任务都有负责人和验收标准。
3. 每次代码变更都通过 PR / MR。
4. 每个 PR 都经过自动化检查和人工审查。
5. 每次发布都可追溯、可回滚、可复盘。
6. AI / Agent 工具只处理标准化辅助工作，不替代人工责任人。

---

## 2. 总体流程

推荐完整流程：

```text
需求提出
  ↓
需求评审
  ↓
任务拆解
  ↓
Issue 创建
  ↓
负责人分派
  ↓
分支创建
  ↓
编码实现
  ↓
本地自测
  ↓
提交 PR
  ↓
自动化检查
  ↓
AI 辅助审查
  ↓
人工 Code Review
  ↓
QA / 验收验证
  ↓
合并
  ↓
自动打包
  ↓
预发布 / 灰度
  ↓
线上发布
  ↓
发布后监控
  ↓
复盘与文档更新
```

该流程可同时适用于 GitHub 和 GitLab：

- GitHub 中使用 Issue / Pull Request / Actions。
- GitLab 中使用 Issue / Merge Request / GitLab CI。
- 核心思想保持一致：任务可追踪、变更可审查、质量可验证、发布可回滚。

## 3. 阶段一：需求提出与评审

### 3.1 输入

需求可能来自：

- 产品规划。
- 策划设计。
- 技术债治理。
- 线上问题。
- QA 反馈。
- 用户反馈。
- 安全漏洞。
- 运维告警。
- 版本计划。

### 3.2 动作

需求提出人应创建 Issue，并补充：

- 背景。
- 目标。
- 业务价值。
- 初步范围。
- 关联资料。
- 期望交付时间。

需求评审时需要确认：

- 是否值得做。
- 是否进入当前版本。
- 是否需要拆分。
- 是否存在依赖。
- 是否需要设计文档或技术方案。
- 是否有明确验收标准。

### 3.3 输出

输出为一个状态明确的 Issue：

- 不做：关闭 Issue，并说明原因。
- 以后做：进入 Backlog。
- 本期做：进入 Ready。
- 需要调研：转为 Research Issue。
- 需要设计：转为 Design / Tech Design Issue。

---

## 4. 阶段二：任务拆解与分派

### 4.1 拆解原则

一个 Issue 应尽量对应一个清晰目标。避免一个 Issue 同时包含多个独立目标。

好的 Issue：

```text
实现背包物品堆叠逻辑。
```

不好的 Issue：

```text
做完整背包系统、UI、存档、联网同步、测试和发布。
```

### 4.2 拆解维度

复杂需求可拆成：

- 设计任务。
- 客户端任务。
- 服务端任务。
- 数据配置任务。
- 工具链任务。
- QA 任务。
- 发布任务。
- 文档任务。

### 4.3 分派要求

每个进入开发的 Issue 必须明确：

- Assignee。
- Priority。
- Milestone。
- Label。
- 验收标准。
- 关联设计或技术文档。

---

## 5. 阶段三：编码实现

### 5.1 创建分支

开发人员从对应基础分支创建工作分支。

工具类、服务类项目：

```text
main
  ↑
feature/<issue-id>-<short-desc>
```

游戏客户端、版本制项目：

```text
develop
  ↑
feature/<issue-id>-<short-desc>
```

示例：

```text
feature/1234-inventory-system
bugfix/1288-login-timeout
```

### 5.2 开发要求

开发过程中应遵守：

- 只修改 Issue 范围内的内容。
- 不扩大任务范围。
- 不顺手做无关重构。
- 不提交密钥、临时文件、构建产物。
- 必要时补充测试和文档。
- 如发现需求不清或范围变化，回到 Issue 中确认。

### 5.3 本地自测

提交 PR 前，开发人员应完成本地验证：

- 编译通过。
- 核心测试通过。
- 关键功能手动验证。
- 日志无明显错误。
- 如有 UI / 游戏表现变化，应附截图、录屏或构建包链接。

---

## 6. 阶段四：提交 PR / MR

### 6.1 PR 的定位

PR 是代码变更进入主线前的唯一入口。PR 不只是合并请求，也是：

- 变更说明。
- 审查入口。
- 自动化验证入口。
- 风险记录。
- 发布追踪依据。

### 6.2 PR 必填内容

PR 应包含：

- 变更摘要。
- 关联 Issue。
- 影响范围。
- 验证方式。
- 截图 / 日志 / 构建产物。
- 已知风险。
- 回滚方案。
- 是否使用 AI 辅助。
- 希望 Reviewer 重点关注的内容。

### 6.3 PR 粒度

建议：

- 一个 PR 只解决一个主要问题。
- 避免超大 PR。
- 大型重构应先有技术方案。
- 功能、重构、格式化尽量拆开。
- 不要把无关改动混入同一个 PR。

---

## 7. 阶段五：自动化检查

### 7.1 触发时机

PR 创建或更新时自动触发：

- Lint。
- 单元测试。
- 构建检查。
- 类型检查。
- CodeQL / 静态分析。
- Dependabot / 依赖检查。
- Secret scanning。
- 游戏项目可加入资源检查、配置表检查、构建 smoke test。

### 7.2 检查结果处理

- 自动化检查失败时，PR 不应合并。
- 开发者负责修复失败项。
- 如果是工具误报，应在 PR 中说明并由负责人确认。
- 对可豁免项，应记录豁免原因和有效期。

### 7.3 自动化检查的定位

自动化检查是质量底线，不是完整质量保证。

它能发现：

- 编译错误。
- 测试失败。
- 常见安全问题。
- 明文密钥。
- 依赖漏洞。
- 风格问题。

它不能替代：

- 架构判断。
- 核心玩法体验。
- 性能权衡。
- 业务逻辑审查。
- 发布决策。

---

## 8. 阶段六：AI 辅助审查

### 8.1 AI 审查的定位

AI 审查用于辅助人工识别常规问题，例如：

- 代码风格不一致。
- 潜在空指针或异常。
- 测试覆盖不足。
- 重复代码。
- 可读性问题。
- PR 摘要和风险提示。

AI 审查不能替代人工审查，尤其不能替代：

- 核心业务逻辑判断。
- 架构设计判断。
- 安全敏感逻辑判断。
- 性能敏感代码判断。
- 游戏体验判断。

### 8.2 推荐触发方式

初期建议使用手动触发：

```text
@claude review this PR
@claude summarize this issue
@claude generate tests for this change
```

不建议一开始对所有 PR 自动触发大量 AI 评论，避免：

- 评论噪音过大。
- API 成本失控。
- Reviewer 忽略真正重要的信息。

### 8.3 AI 输出要求

AI 辅助审查应输出：

- 发现的问题。
- 问题严重程度。
- 涉及文件或代码位置。
- 建议修改方式。
- 是否需要人工决策。
- 不确定之处。

---

## 9. 阶段七：人工 Code Review

### 9.1 Review 重点

Reviewer 应重点关注：

- 是否符合 Issue 目标。
- 是否超出任务范围。
- 是否破坏现有逻辑。
- 是否存在安全问题。
- 是否影响性能。
- 是否有足够测试。
- 是否有回滚方案。
- 是否需要补充文档。
- AI 生成内容是否可靠。

游戏项目还应关注：

- 是否影响核心玩法。
- 是否影响手感、镜头、节奏。
- 是否影响网络同步。
- 是否影响资源加载和内存。
- 是否影响多端一致性。
- 是否影响构建包体和性能指标。

### 9.2 审查人数

建议：

- 普通 PR：至少 1 名 reviewer。
- 高风险 PR：至少 2 名 reviewer。
- 涉及核心模块：必须 Code Owner 审查。
- 涉及发布、权限、安全、支付、数据迁移：必须负责人审批。

### 9.3 合并条件

PR 合并前必须满足：

- 关联 Issue 明确。
- PR 描述完整。
- 必要自动化检查通过。
- 人工 Review 通过。
- 无未解决 blocking comment。
- 高风险变更有回滚方案。
- 如使用 AI，AI 参与范围已说明。

---

## 10. 阶段八：QA / 验收验证

### 10.1 验证内容

QA 或验收人员应根据 Issue 验收标准验证：

- 功能是否符合需求。
- 边界情况是否正常。
- 旧功能是否回归。
- 性能是否异常。
- 日志是否异常。
- UI / 交互是否符合预期。
- 游戏项目是否符合玩法、手感、表现要求。

### 10.2 验证证据

验证结果应回写到 Issue 或 PR：

- 测试环境。
- 测试版本。
- 测试步骤。
- 测试结果。
- 截图 / 录屏 / 日志。
- 遗留问题。

建议使用评论前缀：

```text
[evidence] 验证证据
[blocker] 阻塞事项
[review] 审查意见
[decision] 决策记录
[handoff] 交接总结
```

---

## 11. 阶段九：合并与打包

### 11.1 合并策略

可按项目选择：

- Squash merge：适合保持主线提交清晰。
- Merge commit：适合保留完整分支历史。
- Rebase merge：适合线性历史，但要求团队熟悉 rebase。

建议：

- 普通功能 PR 使用 Squash merge。
- release / hotfix 可保留 merge commit。
- 团队应统一策略，避免混乱。

### 11.2 合并后动作

合并到 `develop` 或 `main` 后自动触发：

- 集成构建。
- 自动部署测试环境。
- 生成构建产物。
- 上传 artifact。
- 通知团队。
- 更新 changelog 或 release draft。

---

## 12. 阶段十：发布

### 12.1 发布前检查

发布前应确认：

- 本版本所有 Issue 状态正确。
- release branch 或 tag 已创建。
- 自动化构建通过。
- QA 验收通过。
- 关键风险已确认。
- 回滚方案已准备。
- 发布说明已完成。
- 负责人已审批。

### 12.2 发布流程

推荐流程：

```text
创建 release branch / tag
  ↓
触发 release workflow
  ↓
构建产物
  ↓
运行发布前检查
  ↓
人工审批
  ↓
发布到测试 / 灰度 / 生产环境
  ↓
发布后验证
  ↓
关闭相关 Issue
```

### 12.3 Hotfix 流程

线上紧急修复应满足：

- 创建 Hotfix Issue。
- 从稳定分支拉 hotfix 分支。
- 最小范围修复。
- 必要自动化检查通过。
- 至少一名负责人审批。
- 发布后补充复盘。
- 修复同步回 develop 或后续开发分支。

---

## 13. 阶段十一：发布后监控与复盘

### 13.1 发布后监控

发布后需要关注：

- 错误日志。
- 崩溃率。
- 性能指标。
- 用户反馈。
- 告警系统。
- 关键业务指标。
- 游戏项目的留存、卡顿、掉线、资源加载异常等指标。

### 13.2 复盘要求

以下情况必须复盘：

- 线上事故。
- 发布回滚。
- 严重缺陷逃逸。
- 安全问题。
- 大型版本发布。
- AI 生成代码引发严重问题。

复盘应记录到：

```text
docs/postmortems/
```

或对应 Incident Issue。

### 13.3 决策沉淀

重要流程或技术决策应沉淀为 ADR：

```text
docs/decisions/0001-use-github-actions-for-ci.md
```

## 14. 工作流角色分工

| 角色 | 主要职责 |
|---|---|
| 需求提出人 | 创建需求 Issue，说明背景和目标 |
| Owner / Leader | 评审需求，确认优先级和范围 |
| Developer | 实现代码，补充测试，提交 PR |
| Reviewer | 审查代码、风险、测试和设计一致性 |
| QA | 验证功能、回归、质量和发布候选版本 |
| DevOps | 维护 CI/CD、发布流程、权限和环境 |
| Release Owner | 负责发布审批、执行和回滚决策 |
| AI / Agent | 辅助生成、审查、总结和自动化，不承担最终责任 |

---

## 15. 工作流落地 Checklist

一个需求从开始到上线，应至少满足：

- [ ] 有对应 Issue。
- [ ] Issue 有明确目标、范围和验收标准。
- [ ] 已分配负责人。
- [ ] 开发分支命名符合规范。
- [ ] PR 关联 Issue。
- [ ] PR 描述完整。
- [ ] 本地自测完成。
- [ ] 自动化检查通过。
- [ ] 必要时完成 AI 辅助审查。
- [ ] 人工 Code Review 通过。
- [ ] QA / 验收通过。
- [ ] 合并到正确分支。
- [ ] 构建和发布成功。
- [ ] 发布后验证完成。
- [ ] Issue 状态更新。
- [ ] 必要文档和复盘已补充。
